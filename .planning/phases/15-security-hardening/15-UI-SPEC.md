# Phase 15 — UI Design Contract

> Visual and interaction contract for Phase 15: Security Hardening.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Scope Note

Phase 15 is primarily backend infrastructure. The **only UI deliverable** is an Audit Log section embedded in the existing admin overview page (`admin/page.tsx`) below the existing Recent Activity feed. CSP headers and the audit logging helper have no visual surface. All design decisions below apply to that single new section.

## Color

Source: `tailwind.config.ts` brand tokens. No shadcn CSS variables — raw hex values used via brand-* utilities.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F8F7F4` (`brand-bg`) | Page background, main content area |
| Secondary (30%) | `#FFFFFF` + `border-gray-100` | Cards (`bg-white rounded-xl border border-gray-100`), sidebar `#111110` (`brand-sidebar`) |
| Accent (10%) | `#1B6B4A` (`brand-accent`) | Active nav item left border + background tint, ADMIN badge, action labels for `user.invite` and `user.role_change` |
| Destructive | `#DC2626` (`brand-red`) | Action label for `user.deactivate` entries in audit log |
| Muted / secondary text | `#5C5C56` (`brand-muted`) | Timestamps, metadata in audit rows |

Accent reserved for: active sidebar nav state (border-l + bg tint), role badge on ADMIN items, and audit log action type indicators for invite and role change actions specifically. Never used for table borders, dividers, or general row hover states.

Positive semantic color: `#16A34A` (`brand-green`) — reserved for `user.reactivate` action label to distinguish from destructive deactivate.

## Interaction States

### Audit Log Loading State

Display `SkeletonText lines={5}` inside the audit section card while data loads — same pattern as the existing activity feed loading state.

### Audit Log Empty State

Render: `<div className="text-sm text-gray-400 text-center py-6">No admin actions recorded yet.</div>`

This matches the exact empty state pattern used in the existing Recent Activity feed.

### Audit Log Error State

Handled by `SectionErrorBoundary`. When the audit log fetch fails, the boundary renders the fallback using existing `common.error_section` and `common.error_section_body` i18n keys. The rest of the admin overview (stats, recent activity) remains functional.

### No Interactive Elements in Audit Log

The audit log table is read-only — no filtering, no pagination, no sort controls in v1. Display most recent 50 entries as a static list per D-11. No click targets, no expand rows.

## Copywriting Contract

Source: CONTEXT.md specifics + established en.json admin namespace patterns.

| Element | Copy (EN) | Copy (ES) |
|---------|-----------|-----------|
| Audit log section heading | "Audit Log" | "Registro de Auditoría" |
| Empty state | "No admin actions recorded yet." | "Aún no hay acciones de administrador registradas." |
| Error state (boundary fallback) | Uses existing `common.error_section` / `common.error_section_body` | Same keys (already translated) |
| Action: user.invite | "{actor} invited {target} as {role}" | "{actor} invitó a {target} como {role}" |
| Action: user.role_change | "{actor} changed {target}'s role to {role}" | "{actor} cambió el rol de {target} a {role}" |
| Action: user.deactivate | "{actor} deactivated {target}" | "{actor} desactivó a {target}" |
| Action: user.reactivate | "{actor} reactivated {target}" | "{actor} reactivó a {target}" |

Human-readable descriptions per D-11 and CONTEXT.md specifics: never display raw action codes (e.g. "user.invite") in the UI.

### Destructive Actions in This Phase

No destructive actions are exposed in the UI for Phase 15. The audit log is read-only. Admin deactivation/reactivation actions that are **logged** originate from the Users page (Phase 12) — audit log just displays them.

CSP headers and the `logAuditEvent()` helper have no user-facing confirmation flows.
