# Phase 3 — UI Design Contract: Detail Pages

> Visual and interaction contract for Phase 3 detail pages.
> Generated by gsd-ui-researcher. Consumed by gsd-planner and gsd-executor.
>
> **Governing principle:** Detail pages are NOT a new design system. They inherit every token,
> component, and pattern from `tokens.css`, `base.css`, and `landing.css` established in
> Phases 1 and 2. This contract documents how the existing system extends to detail-page
> layouts — specifically the new sections (hero override, deliverables, timeline, outcomes,
> values) and what CSS goes into the new `detail.css` file.

## Spacing Scale

All spacing uses tokens from `tokens.css`. No raw px values in new CSS.

| Token | Value | Usage in detail pages |
|-------|-------|-----------------------|
| `--space-1` | 4px | Inline gaps, badge padding |
| `--space-2` | 8px | Compact list item gaps, tight padding |
| `--space-3` | 12px | Card internal top/bottom |
| `--space-4` | 16px | Default element spacing, icon-to-text gap |
| `--space-5` | 20px | Left-padding on deliverable list items |
| `--space-6` | 24px | Card padding, section sub-element spacing |
| `--space-8` | 32px | Deliverable item bottom margin |
| `--space-10` | 40px | Section intro paragraph bottom margin |
| `--space-16` | 64px | Section header bottom margin |
| `--space-20` | 80px | Section vertical padding (matches `--section-padding-y`) |

Exceptions:
- Nav height: 64px via `--nav-height` (existing token, not a new exception)
- Detail page hero: `padding: var(--space-20) 0 var(--space-16)` — replaces the full-viewport `min-height: calc(100vh - var(--nav-height))` used on index.html
- Timeline connector line: 2px width (structural, not spacing — use absolute positioning)
- Touch targets for CTA buttons: minimum 44px height (btn--lg achieves this via existing base.css padding)

## Typography

Inherit all scales from `tokens.css`. Detail pages use exactly 4 roles with these assignments:

| Role | Token | Computed Size | Weight Token | Weight | Line Height Token | Line Height | Usage |
|------|-------|--------------|--------------|--------|-------------------|-------------|-------|
| Body | `--text-base` | 16px | `--font-normal` | 400 | `--leading-normal` | 1.6 | Deliverable body, card descriptions, timeline step body, manifesto items, section intro paragraphs |
| Label | `--text-xs` | 12px | `--font-bold` | 700 | `--leading-none` | 1 | Pill/tag badges, section labels (MISSION, VISION, VALUE 01), mono badges |
| Heading | `--text-xl` | 20px | `--font-bold` | 700 | `--leading-snug` | 1.375 | Deliverable titles, outcome card titles, timeline step titles, value card titles, mission/vision/purpose block headings |
| Display | `--text-4xl` | 36px | `--font-bold` | 700 | `--leading-tight` | 1.15 | Page hero headline (`h1`) on detail pages |

Section headings use the `.section-title` class (existing in base.css). Hero subheadline uses body role sizing (`--text-base`, 16px). Price amounts follow the existing pricing pattern in landing.css.

Font family rules:
- All body and heading text: `--font-sans` (DM Sans)
- All pill/tag badges, numeric indexes (01–06), price amounts: `--font-mono` (JetBrains Mono)

## Color

Inherit all values from `tokens.css`. No new colors introduced in Phase 3.

| Role | Token | Hex | Usage in detail pages |
|------|-------|-----|-----------------------|
| Dominant (60%) | `--color-bg` | `#09090b` | Page background — `<body>`, hero section background |
| Secondary (30%) | `--color-surface` | `#18181b` | Deliverable cards, outcome cards, value cards, alternate section backgrounds |
| Surface elevated | `--color-surface-elevated` | `#27272a` | Hero dot-grid dots, hover states on cards |
| Accent (10%) | `--color-accent` | `#3b82f6` | Reserved for specific elements only (see below) |
| Destructive | `--color-danger` | `#ef4444` | Not used in Phase 3 — no destructive actions exist |

Accent (`--color-accent` #3b82f6) is reserved for ONLY these elements:
1. Hero headline accent span (`.page-hero__headline-accent`) — the key phrase in italic or emphasis
2. Primary CTA button background (`btn--primary` class — already defined in base.css)
3. Focus rings on interactive elements (`--color-border-focus` token)
4. The `.cta-section` gradient endpoint (`--color-accent-subtle` background)
5. The `.cta-section` top border (`--color-accent-border`)

Service-specific accent colors (inherited from landing.css pattern, applied on detail pages):
- Observability pages: `--color-success` (#22c55e green) for top card border, tag label, check marks
- Fractional SRE pages: `--color-warning` (#f59e0b amber) for top card border, tag label, check marks
- Mission page: `--color-accent` (#3b82f6 blue) for section labels and numeric indexes

Timeline connector dot color: `--color-accent` (#3b82f6) — consistent with success criterion that each detail page has a clear visual anchor.

Text colors:
- Primary text: `--color-text` (#fafafa) — all headings, price amounts
- Secondary text: `--color-text-muted` (#a1a1aa) — body paragraphs, card descriptions, timeline step bodies
- Tertiary/meta text: `--color-text-subtle` (#71717a) — price period, section sublabels

### Empty States

No data-loading UI exists on these pages. All content is static. Empty state not applicable.

Exception: if a detail page's i18n key is missing from one language file, the i18n system returns the raw key string as visible text. This is the de facto "empty state" — it looks like `obs_page.hero.headline` in the DOM. Prevention: all keys must be added to both `en.json` and `es.json` in the same commit. No special UI treatment needed.

### Error States

No form submission, API calls, or dynamic data on these pages. The only error scenario is:
- **Calendly widget not yet loaded when CTA clicked:** The existing fallback in index.html (`window.open(CALENDLY_URL, '_blank', 'noopener,noreferrer')`) must be copied verbatim to all detail pages. No additional error UI needed.
- **Partial fetch failure (nav/footer 404):** If `VANTIX_BASE` is wrong, nav/footer divs render empty. No error message to the user — prevention via correct `window.VANTIX_BASE = '..'` is the fix.

### Destructive Actions

None. Phase 3 has no destructive user actions. No confirmation dialogs, no delete patterns.

## Page Layouts and Section Contracts

This section is the primary new content of the Phase 3 UI-SPEC. It prescribes the layout structure for each new section type that does not exist in `landing.css`.

### All Detail Pages: HTML Shell

Every detail page uses this exact shell. Deviation from this template is a defect.

```
<!DOCTYPE html> → <html lang="en"> → <head>
  Charset, viewport, description, title
  Google Fonts preconnect (fonts.googleapis.com + fonts.gstatic.com crossorigin)
  DM Sans + JetBrains Mono Google Fonts link
  ../css/tokens.css
  ../css/base.css
  ../css/detail.css           ← NEW in Phase 3
  Calendly widget.css
  GA4 snippet (same G-XXXXXXXXXX placeholder as index.html)
</head>
<body>
  <div data-partial="nav"></div>
  <main id="main-content" tabindex="-1">
    [page sections]
  </main>
  <div data-partial="footer"></div>
  <script>window.VANTIX_BASE = '..';</script>   ← CRITICAL: '..' not '.'
  <script src="../js/i18n.js"></script>
  <script src="../js/partials.js"></script>
  Calendly widget.js (async)
  Calendly popup trigger script (identical to index.html)
  GA4 booking event listener (identical to index.html)
</body>
```

Source: RESEARCH.md Pattern 1 (derived from index.html audit)

### New CSS File: `css/detail.css`

All new CSS for detail pages goes into this single file. It must follow the same rule as all other CSS files: no raw hex values, all values via token variables.

Sections to define in `detail.css`:

1. `.page-hero` — detail page hero override
2. `.deliverables` — numbered deliverable items with tool/frequency badges
3. `.outcomes-grid` — 3-column outcome card grid
4. `.timeline` — vertical timeline for "How it works"
5. `.values-grid` — 2-column values card grid (mission page)
6. `.manifesto` — manifesto list (mission page)
7. `.mission-block`, `.vision-block`, `.purpose-block` — mission page content blocks

### Section 1: Page Hero (`.page-hero`)

Used on all three detail pages. Replaces `.hero` from landing.css.

| Property | Value | Rationale |
|----------|-------|-----------|
| Background | `var(--color-bg)` | Same as page background — no separate section color |
| Padding | `var(--space-20) 0 var(--space-16)` | 80px top, 64px bottom — shorter than full-viewport landing hero |
| Min-height | none — do NOT use `calc(100vh - var(--nav-height))` | Detail pages must show content below the fold |
| Dot-grid background | Same `::before` pseudo as `.hero` in landing.css — radial-gradient of `--color-surface-elevated` at 1px, 24px × 24px, opacity 0.6 | Visual consistency with index.html |
| Accent glow | Same `::after` pseudo as `.hero` in landing.css — 900px × 700px radial ellipse from top center using `--color-accent-glow` | Visual consistency |
| Max-width of inner | 780px (matches `.hero__inner`) | Consistent reading width |
| Pill/tag | `--font-mono`, `--text-xs`, `--tracking-wider`, uppercase | Same as `.hero__pill` but color varies by service (see color section) |
| Headline (`h1`) | `--text-4xl` (36px), weight 700, `--leading-tight` | Smaller than landing `--text-6xl` — detail pages are deep eval, not first impression |
| Subheadline | `--text-base` (16px), weight 400, `--leading-normal`, color `--color-text-muted` | Body role sizing — same visual weight as `.hero__sub` |
| Max-width subheadline | 600px | Maintains readable line length |

Mobile breakpoints:
- `max-width: 768px`: headline drops to `--text-xl` (20px, heading role)
- `max-width: 480px`: hero actions: row → column, buttons full-width

## Content Decisions

### Pricing on Detail Pages

- Canonical prices from `en.json` / index.html are authoritative.
- Observability page: show `$3,000/month` (from `services.observability.price`)
- Fractional SRE page: show `$5,500/month` (from `services.sre.price`)

### Content That Needs Human Translation

- All `en.json` keys under `obs_page.*` — the Spanish content from the legacy file is authoritative; English must be authored (not machine-translated).
- All `en.json` keys under `mission_page.*` — mission, vision, purpose, values, and manifesto are brand-voice content requiring human authorship.
- All `sre_page.*` keys in both languages — net-new content; Spanish and English must be authored following the observability page structure.

These are content dependencies, not code blockers. The executor must flag these as "content to be provided" in i18n keys where translation is pending.

## Anti-Patterns (Hard Constraints for Executor)

| Anti-Pattern | Why It Fails | Correct Pattern |
|-------------|--------------|-----------------|
| `window.VANTIX_BASE = '.'` on detail pages | Causes 404 for all partials and i18n JSON files | `window.VANTIX_BASE = '..'` |
| Inline `<style>` tags | Violates DSNG-01 (no raw values outside tokens system) | External `detail.css` only |
| Hard-coded text without `data-i18n` | Text never switches language | Every visible string needs `data-i18n` |
| `<a href="https://calendly.com/...">` for CTAs | Navigates away from page; breaks GA4 tracking | `<button class="js-calendly-trigger">` |
| Raw hex values in CSS | Violates DSNG-01 | CSS Custom Properties from `tokens.css` |
| Landing page hero class `.hero` on detail pages | `min-height: calc(100vh - nav-height)` hides content below fold | Use `.page-hero` with padding-based height |
| Per-page CSS files | Fragmentation | Single `detail.css` for all detail pages |
| Copying i18n keys without adding to both files | Key visible as raw dot-notation in one language | Always commit keys to `en.json` and `es.json` together |
