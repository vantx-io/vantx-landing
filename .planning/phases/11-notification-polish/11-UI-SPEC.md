## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge padding, toggle dot inset |
| sm | 8px | Compact element spacing, toggle track padding |
| md | 16px | Default element spacing, section gaps |
| lg | 24px | Section padding, settings card padding |
| xl | 32px | Main content padding (p-8 = 32px on portal main) |
| 2xl | 48px | Major section breaks within settings page |
| 3xl | 64px | Page-level spacing (not used in this phase) |

Exceptions:
- Toggle track: 44px wide × 24px tall — this is a component dimension (touch target size per WCAG 2.5.5), not a spacing scale token. It does not violate the 8-point scale.

Note: The existing sidebar uses `px-5 py-2.5` and `p-5` patterns inherited from prior phases. This phase does not modify those values and does not bring them into scope here.

Source: portal/layout.tsx (p-8), accessibility skill WCAG 2.5.5 touch target

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 (regular) | 1.5 |
| Label | 13px | 400 (regular) | 1.4 |
| Heading (section) | 16px | 600 (semibold) | 1.2 |
| Small / meta | 11px | 400 (regular) | 1.4 |

Two weights only: 400 (regular) and 600 (semibold).

Portal UI uses `text-sm` (14px) as the primary body size. Section headings within settings cards use `text-base font-semibold` (16px). Nav labels and toggle labels use `text-[13px]`. Timestamps and helper text use `text-[11px]`.

The 4-size / 2-weight constraint above applies only to portal UI components. Email typography operates under a separate scale, inheriting the TaskStatusEmail D-09 pattern and is not subject to this constraint.

**Email typography (WeeklyDigestEmail.tsx only — separate scale, inherits TaskStatusEmail D-09 pattern):**

- Body: 15px, line-height 1.6 — matches TaskStatusEmail pattern (D-09)
- Footer: 12px — matches TaskStatusEmail pattern (D-09)

## Component Inventory

### New Components (this phase)

#### 1. Toggle Switch (`ToggleSwitch`)
A custom Tailwind toggle, hand-rolled. No external library (RESEARCH.md: project avoids external component libs).

**Anatomy:**
- Track: `w-11 h-6 rounded-full` (44px × 24px) — satisfies WCAG 2.5.5 touch target
- Dot: `w-5 h-5 rounded-full bg-white shadow` — translates on toggle
- ON state: track `bg-brand-accent` (green), dot translated to right (`translate-x-5`)
- OFF state: track `bg-[#D1D5DB]` (gray-300), dot at left (`translate-x-0`)
- Transition: `transition-all duration-200` on track, `transform transition-transform duration-200` on dot
- Focus ring: `focus:outline-none focus-visible:ring-2 focus-visible:ring-brand-accent focus-visible:ring-offset-2` — WCAG 2.4.7
- ARIA: `role="switch"`, `aria-checked={enabled}`, `aria-label={label}` on the `<button>` element
- Disabled (during API call): track `opacity-60 cursor-not-allowed`, dot movement halted
- No loading spinner — the transition animation is the only feedback (D-13: instant visual feedback)

**Interaction contract:**
1. User clicks toggle
2. Local state flips immediately (optimistic update) — track color changes instantly
3. PATCH request fires in background (`/api/preferences`)
4. On API failure: local state rolls back to previous value after ~300ms debounce
5. No toast/notification on success — the visual toggle state IS the confirmation

#### 2. Settings Page (`/portal/settings/page.tsx`)
Client component. Follows existing portal page pattern (`"use client"`, `useEffect` for initial load).

**Primary focal point:** the Notifications section card — the sole content block on the settings page.

**Layout:**
- Page heading: `text-xl font-semibold text-white` at top (adapts to portal main area `p-8`)
- Settings section card: `bg-brand-surface rounded-xl border border-gray-200 p-6`
- Section heading: `text-base font-semibold text-[#1A1A17] mb-4`
- Toggle row: `flex items-center justify-between py-3 border-b border-gray-100 last:border-0`
- Toggle label: `text-sm text-[#1A1A17]`
- Toggle description: `text-[11px] text-brand-muted mt-0.5`

**Three toggle rows (in order, per D-12):**
1. "Email notifications" / "Notificaciones por email" — controls `email_enabled`
2. "In-app notifications" / "Notificaciones en la app" — controls `in_app_enabled`
3. "Weekly digest email" / "Resumen semanal por email" — controls `digest_enabled`

**Loading state:**
- On initial load (fetching preferences): render three toggle rows with toggles disabled and `opacity-60`
- No skeleton placeholder in this phase — Phase 14 (UX-01) handles universal loading skeletons
- Once data resolves, toggles become interactive

#### 3. Settings Nav Item (sidebar addition)
Add to `navKeys` array in `portal/layout.tsx` as the last entry:
```
{ id: 'settings', key: 'settings', segment: '/settings' }
```
Active/hover styles: inherits existing nav item class logic (`border-brand-accent bg-brand-accent/15 text-white font-semibold` when active, `border-transparent hover:bg-white/5 hover:text-white/80` otherwise).

Icon: Add `Settings` icon from `lucide-react` to the nav item (16px, `w-4 h-4`) to distinguish Settings from other nav items that currently have no icons. Position: `flex items-center gap-2.5`.

#### 4. WeeklyDigestEmail (`WeeklyDigestEmail.tsx`)
React Email component. Follows TaskStatusEmail layout exactly (D-09).

**Structure:**
- `Html > Head > Preview > Body > Section`
- Max-width: 560px, centered, white card on `#f9f9f9` background
- Padding: 32px on the Section container
- Heading: `fontSize: 22, fontWeight: 'bold', color: '#111'`
- Subheading ("Your week at a glance" / "Tu semana de un vistazo"): `fontSize: 16, color: '#333'`
- Status counts table: three cells (New / In Progress / Completed) rendered as inline blocks, `fontSize: 14, color: '#444'`
- Task list (max 10): `fontSize: 13, color: '#444'`, each task on its own line with status label
- Truncation line ("and N more" / "y N más"): `fontSize: 13, color: '#888', fontStyle: 'italic'` — only rendered when `totalTasks > 10`
- CTA Button: `background: '#2563EB', color: '#fff', padding: '12px 24px', borderRadius: 6` — label "Go to portal" / "Ir al portal"
- Divider: `<Hr />` before footer
- Footer: `fontSize: 12, color: '#888'` — "Vantx · vantx.io · hello@vantx.io"

## Interaction States

### Toggle Switch States

| State | Visual |
|-------|--------|
| ON (enabled) | Track: `bg-brand-accent` (#1B6B4A), dot at right |
| OFF (disabled) | Track: `bg-gray-300` (#D1D5DB), dot at left |
| Hover (any state) | Track: `opacity-90`, cursor `pointer` |
| Focus-visible | 2px ring `brand-accent` with 2px offset — keyboard nav |
| Saving (in-flight) | Track: `opacity-60`, cursor `not-allowed`, no interaction |
| Error (rolled back) | Track snaps back to previous state; error message appears below row |

### Settings Page States

| State | Visual |
|-------|--------|
| Loading (initial fetch) | All three toggles rendered but `opacity-60 pointer-events-none` |
| Loaded | Toggles interactive, reflect DB values |
| Save error | Inline error text `text-[11px] text-brand-red` below affected toggle |
| Page fetch error | Error message `text-sm text-brand-red` below section heading |

## Accessibility Requirements

Source: `.agents/skills/accessibility/SKILL.md`

| Requirement | Implementation |
|-------------|----------------|
| Toggle WCAG 2.5.5 touch target | Toggle track 44px × 24px minimum |
| Toggle keyboard accessible | `role="switch"` `aria-checked` on `<button>`, operable via Space/Enter |
| Focus visible | `focus-visible:ring-2 focus-visible:ring-brand-accent` on toggle button |
| Toggle label association | Toggle `aria-label` matches visible label text, or use `aria-labelledby` pointing to visible label |
| Color not sole indicator | Toggle ON/OFF also indicated by dot position (not color alone) |
| Error announcement | Error messages use `role="alert"` so screen readers announce them on appearance |
